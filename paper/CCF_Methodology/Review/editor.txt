*Please note, Data Descriptors should not include significant results, conclusions, discussion or data analyses. The goal is to share data for others to use, with the details in the paper intended to provide transparency (how data were collected or validated) or practical help (how to find the data, what variables/column headings mean, etc.) We do not cover novel methodologies, frameworks, or cover trend analysis as part of this scope. If you would like to continue with this work, you will need to revise all sections of the manuscript to outline the dataset that is meant to be shared with this Data Descriptor that readers will be able to use as a standalone data product in their own research

* Because at least some of your data looks to originate from documents or literature, we request some standard information. Firstly, please confirm the specific location (paper) is listed against each data point in the data file, with a variable or column heading such as ‘References’ or ‘Sources’ pointing users to the input data. Permanent digital identifiers (e.g. DOIs) are much preferable to written references, so do not use imprecise formats such as “Jones et al, 2025”, or place data sources a separate list or tab. All data sources must have a source, so if there are “unpublished” data being listed then these need to be removed or treated as primary data where the methods are listed directly in the text. You may also add a general list of literature references to the main article, if there are fewer than 200, as this provides additional credit to the data owner. We do not recommend authors of cited works are listed as co-authors, if the data are previously published, and where the individuals have not otherwise contributed (in the same way review articles use citation, not co-authorship, to credit prior art). Works do not have to be peer reviewed to use (e.g. pre-prints, theses, etc) if you have validated their accuracy. Finally, check your text or data extraction is within fair use. This allows re-distribution without direct permission for small snippets, facts, or numerical data, but not full narrative text, (original) images, or any other item that may breach copyright which you may need direct permission from the data owner to override – unless the item is explicitly openly licensed (e.g. CC-BY).

* Please check the Methods section to ensure the content is in scope. This should solely focus on what was practically done to create the data, written in the past tense, to describe completed processes or tasks. It should be written with enough detail to allow someone to reproduce the same dataset using the same inputs and instructions, including all data collection, creation, curation or processing steps, to produce the same result. Code and input data should also be described and supplied to allow this, citing all sources where applicable, in addition to elements such as data obtained by API (e.g. which end points where used). Where existing tools, methods, background or theory are used, these should be described to explain which elements were used, but it may be possible to cite the details via the original works, rather than copy. i.e. the section does not need to explain how an existing theory or methods work, unless crucial, on the understanding an expert audience can read the information elsewhere. For data obtained via agreements with third parties, details should be provided on who those parties were and how to contact them.
